01

Handling & safety

The board is more robust than it looks, but a handful of habits prevent the failures that actually happen — static discharge, shorts, and the SD card issue you already ran into.

Static electricity (ESD)

Your body holds a static charge you can't feel — often thousands of volts — and the chips on the board can be damaged by a fraction of that. It doesn't spark or hurt; the chip just quietly dies or starts glitching.

  • Touch a grounded metal object (a radiator, an unpainted PC case) before picking the board up.
  • Hold the board by its edges — avoid touching the chips, the GPIO pins, or the underside components.
  • Work on a hard, non-carpeted surface if possible. Carpet + dry air is the worst combination for static buildup.

Shorts

avoid

Never set a powered board down on a metal surface, a conductive antistatic bag's metallic side, or anything with loose wires nearby. A single stray strand of wire bridging two pins can short a rail and kill the board or your power supply instantly.

The SD card issue (your specific case)

Covered in depth already — the takeaway for this poster: the microSD slot on a 2B is a plain friction connector with no lock. Treat the card as sealed inside the case once it's imaged and working. Any physical build (like your cyberdeck) should route the slot so nothing external can press on it — see Section 03.

Power discipline

  • Always power up last. Connect SD card, GPIO wiring, HDMI, keyboard, everything else first — power cable goes in last.
  • Never hot-plug GPIO wiring. Connecting/disconnecting sensors or wires to GPIO pins while the board is powered can short pins together mid-insertion. Power down first.
  • Don't just pull the power cable to turn it off — this is the same risk as the SD card issue but for the whole filesystem. Use sudo shutdown -h now and wait for the green activity LED to stop blinking before cutting power.
02

Board anatomy

The five zones from the diagram above, expanded with what each part actually is and why it's shaped the way it is.

SoC — system on chip

One chip containing the CPU (does the computing), GPU (handles graphics/video), and memory controller. Everything "the Pi does" happens inside this one piece of silicon.

RAM

Working memory. Stacked on top of the SoC on most models (package-on-package) — invisible as a separate chip. Wiped completely on power loss; this is where your running programs and OS state live while active.

microSD slot

The only persistent storage and, on older Pis, the only place the boot ROM knows to look. Holds the bootloader, OS, and your files.

USB ports

For keyboard, mouse, storage, and other peripherals. Also used to power the board via a USB power supply (separate from data USB ports on some models).

Ethernet port

Wired networking — more reliable and lower-latency than Wi-Fi, useful when headless (no monitor) and doing remote work over SSH.

HDMI out

Digital video + audio to a monitor or TV. Some smaller Pi models use mini/micro-HDMI — check your board before buying a cable.

Audio jack

Analog audio out on boards that have one (not all do — some rely on HDMI audio or USB audio only).

GPIO header

40 pins of raw electrical access to the SoC. Digital in/out, plus dedicated pins for power, ground, and communication protocols. Covered fully in Section 06.

The boot chain, concretely

  1. Power applied — the SoC's boot ROM (etched at the factory, unchangeable) wakes up.
  2. ROM looks for a bootloader — on the SD card (older Pis), or SD/USB/network depending on model and configured boot order (newer Pis).
  3. Bootloader loads the kernel — a small program (bootcode.bin / start.elf on older Pis, or firmware built into newer ones) reads config files and hands off to the Linux kernel.
  4. Kernel initializes hardware — sets up RAM, drivers, filesystems.
  5. init/systemd starts userspace — the actual OS you interact with: services, networking, and eventually your login prompt or desktop.
03

Assembly order

A sane build order for a project like your cyberdeck — minimizes rework and avoids powering the board before it's safe to.

  1. 1
    Image the SD card first, on your laptop
    Do this before any physical assembly — see Section 04. Test that it boots on the bare board (just power + HDMI + keyboard) before it goes inside an enclosure.
  2. 2
    Mount the board to standoffs or a plate
    Use brass standoffs/screws through the board's corner mounting holes — never let the board rest directly on a conductive surface.
  3. 3
    Route and secure the SD card
    If using a right-angle extender or recessed mount, install and test this before closing the case — SD failures are much easier to diagnose with the board accessible.
  4. 4
    Wire GPIO peripherals with the board unpowered
    Screen, buttons, sensors — everything connects while power is disconnected. Double check polarity on anything with a + / − before first power-up.
  5. 5
    Close the enclosure, leaving ports accessible
    Leave USB, power, and any debug headers reachable without disassembly — you'll want them during software setup.
  6. 6
    First power-up, watch the LEDs
    Red = power good. Green blinking = SD card activity (a good sign — it's reading the boot files). No green activity at all usually means an imaging or SD contact problem.
for your build specifically

Since you're on a Pi 2B with a friction SD slot: image and fully verify the card works before it goes anywhere near the final enclosure, and clone a backup image immediately once it's working (Section 04 covers this). That turns "case pressed on the card, corrupted everything" from a full rebuild into a five-minute reflash.

04

Imaging the OS

Getting an operating system onto the SD card, and setting it up so it can boot without a monitor attached (headless).

Tool: Raspberry Pi Imager

The official tool (raspberrypi.com/software). Handles downloading the OS image, writing it to the card, and pre-configuring Wi-Fi/SSH/username before the card ever touches the Pi.

  1. 1
    Choose OS
    Raspberry Pi OS Lite (no desktop — smaller, faster boot, ideal for headless/embedded projects like a cyberdeck) or Raspberry Pi OS with Desktop if you want a GUI.
  2. 2
    Choose storage
    Select your SD card. Double-check you've picked the right device — this step erases everything on the target.
  3. 3
    Edit settings (gear icon) before writing
    Set hostname, enable SSH, set a username/password, and pre-fill Wi-Fi SSID/password. This is what makes headless setup possible — no monitor needed at all on first boot.
  4. 4
    Write, then verify
    Imager writes and verifies automatically. Don't remove the card until it says done.

Backing up a working card (do this after every milestone)

# macOS / Linux — find the disk first with `diskutil list` or `lsblk`
sudo dd if=/dev/diskN of=~/pi-backup.img bs=4M status=progress

# Restore later:
sudo dd if=~/pi-backup.img of=/dev/diskN bs=4M status=progress

On Windows, Raspberry Pi Imager can write a custom .img file back to a card, or use Win32DiskImager to both read and write images.

Headless first connection

With SSH enabled and Wi-Fi pre-configured in Imager, boot the Pi with no monitor attached and connect from your laptop:

ssh username@hostname.local
# e.g. ssh sabrine@raspberrypi.local
tip

If .local hostname resolution doesn't work on your network, find the Pi's IP from your router's device list, or run a network scan (arp -a, or an app like Fing) and SSH to the IP directly.

07

Connecting a screen

Three genuinely different paths depending on what kind of display you're using — they are not interchangeable in how you configure them.

HDMI monitor

Plug and play. The Pi outputs a full framebuffer over HDMI automatically — no config needed for a standard monitor or TV.

SPI display (small GPIO screens)

Small (often 1.5–3.5") screens wired directly to GPIO pins using the SPI protocol. Needs a driver enabled and, usually, a specific overlay for that exact display model.

DSI display (official touchscreen)

Connects via the dedicated DSI ribbon cable port, not GPIO. Usually detected automatically by recent Raspberry Pi OS versions.

Setting up a small SPI/GPIO screen

This is the path most cyberdeck-style builds use, so it's worth doing in detail. The general flow:

  1. 1
    Wire the screen to the correct pins
    Typically: VCC → 3.3V, GND → Ground, plus SPI pins (MOSI, SCLK, CE0) and a couple of GPIO control lines (DC/data-command, RST/reset, sometimes backlight). The exact pins depend on your specific screen model — check its datasheet.
  2. 2
    Enable SPI
    sudo raspi-config → Interface Options → SPI → enable, then reboot.
  3. 3
    Install the driver
    Most small SPI screens use a common controller chip (ILI9341, ST7789, etc). Install the matching driver — often via a manufacturer's install script, or a Python library like luma.lcd or Adafruit's CircuitPython display drivers.
  4. 4
    Choose console framebuffer vs your own drawing
    You can mirror the whole Linux console/desktop onto the small screen (a framebuffer driver), or — more common for custom projects — leave the console on HDMI and draw directly to the SPI screen from your own Python script using a graphics library.

Drawing to the screen from Python

from luma.core.interface.serial import spi
from luma.lcd.device import st7789
from PIL import ImageDraw

serial = spi(port=0, device=0, gpio_DC=24, gpio_RST=25)
device = st7789(serial, width=240, height=240)

with device:
    pass  # context manager clears the screen

draw_img = device.display  # or build a PIL Image and call device.display(image)

Libraries like luma.lcd and Adafruit's CircuitPython display drivers both hand you a PIL (Python Imaging Library) canvas — meaning once the screen is talking, you draw to it with normal image/text drawing calls, same as you would to any image file.

08

Programming the GPIO

Every practical way people actually write code to control pins, roughly ordered from easiest to most advanced.

Python — gpiozero (recommended starting point)

A high-level library built specifically to make common components (LEDs, buttons, motors, sensors) trivial to use, hiding the raw pin-toggling.

from gpiozero import LED, Button
from time import sleep

led = LED(17)      # BCM numbering
button = Button(2)

button.when_pressed = led.on
button.when_released = led.off

while True:
    sleep(1)

Python — RPi.GPIO (lower level, still very common)

Older, more explicit library. You manage pin modes and states yourself — closer to what's actually happening electrically.

import RPi.GPIO as GPIO
import time

GPIO.setmode(GPIO.BCM)
GPIO.setup(17, GPIO.OUT)

try:
    while True:
        GPIO.output(17, GPIO.HIGH)
        time.sleep(1)
        GPIO.output(17, GPIO.LOW)
        time.sleep(1)
finally:
    GPIO.cleanup()   # always release pins on exit

C — pigpio / WiringPi-successor libraries

For timing-critical work (precise PWM, fast polling) Python's overhead can matter. pigpio runs a background daemon and gives C, Python, and even network-socket access to precise pin timing.

#include <pigpio.h>

int main() {
    gpioInitialise();
    gpioSetMode(17, PI_OUTPUT);
    gpioWrite(17, 1);
    gpioTerminate();
    return 0;
}
// compile: gcc file.c -lpigpio -lrt -pthread -o out

Shell — for quick tests, no code needed

# Using the modern gpio character device interface:
gpioset gpiochip0 17=1     # set GPIO17 high
gpioget gpiochip0 17       # read GPIO17's state

Node.js — onoff

const Gpio = require('onoff').Gpio;
const led = new Gpio(17, 'out');

led.writeSync(1);
setTimeout(() => { led.writeSync(0); led.unexport(); }, 1000);

Which one to actually reach for

SituationUse
Learning, quick prototypes, most hobby projectsgpiozero
Need explicit low-level control, older tutorialsRPi.GPIO
Precise timing (audio, fast PWM, stepper motors)pigpio
Integrating with a Node.js app/serveronoff or pigpio's Node bindings

always

Release pins cleanly on exit (GPIO.cleanup(), led.unexport(), etc). Skipping this leaves pins in whatever state they were left, which can confuse the next run of your script or another program trying to use the same pin.

09

Protocols: I2C · SPI · UART · PWM

Plain digital GPIO can only be on or off. These four are the standard ways the Pi talks to more complex components — each solves a different communication problem.

I2C — two wire, many devices

Two shared wires (SDA/data, SCL/clock) let you connect many devices to the same two pins, each addressed by a unique ID. Common for sensors (temperature, IMUs), small OLED screens, real-time clocks. Slower but very wire-efficient.

SPI — fast, one device per chip-select

Four+ wires (MOSI, MISO, SCLK, CE/chip-select), much faster than I2C. Each device needs its own chip-select line. Used for displays, SD card readers, and anything needing higher data throughput.

UART — simple serial, point to point

Two wires (TX/RX), the oldest and simplest serial protocol. Used for direct communication with a single other device — GPS modules, some sensors, or debugging another microcontroller.

PWM — simulating analog output

Rapidly switches a pin on/off to approximate a variable voltage. Used for LED brightness, motor speed, and servo control — the Pi has no true analog output, so PWM is the standard workaround.

Enabling each protocol

sudo raspi-config
# → Interface Options → I2C / SPI / Serial Port(UART) → enable each you need
# reboot after enabling

I2C — scanning for connected devices

sudo apt install i2c-tools
i2cdetect -y 1
# prints a grid of addresses — a number shown means a device answered there

I2C in Python

from smbus2 import SMBus

bus = SMBus(1)
data = bus.read_byte_data(0x68, 0x00)  # device address, register

PWM in Python (gpiozero)

from gpiozero import PWMLED
from time import sleep

led = PWMLED(18)
for cycle in range(3):
    led.value = 0.1   # 10% duty cycle — dim
    sleep(1)
    led.value = 1.0   # 100% — full brightness
    sleep(1)

Choosing which protocol a component needs

You don't choose this — the component's datasheet does. Check what the part actually exposes: if it lists "SDA/SCL" it's I2C, "MOSI/MISO/SCK/CS" it's SPI, "TX/RX" it's UART. Match your wiring and code library to what's printed on the datasheet, not the other way around.

Software stack & services

Once code runs once, the next question is making it run reliably — on boot, in the background, without a terminal attached.

Running your script automatically on boot — systemd (recommended)

sudo nano /etc/systemd/system/mydeck.service
[Unit]
Description=My cyberdeck app
After=network.target

[Service]
ExecStart=/usr/bin/python3 /home/sabrine/deck/main.py
WorkingDirectory=/home/sabrine/deck
Restart=on-failure
User=sabrine

[Install]
WantedBy=multi-user.target
sudo systemctl enable mydeck.service
sudo systemctl start mydeck.service
sudo systemctl status mydeck.service   # check it's running
journalctl -u mydeck.service -f        # live log output

This is the production-grade way to do it — the service restarts automatically if your script crashes, starts on every boot, and its logs are centrally viewable.

Quick alternative — cron

crontab -e
# add: @reboot python3 /home/sabrine/deck/main.py &

Simpler but cruder — no automatic restart on crash, harder to monitor. Fine for quick tests, not for anything you're relying on.

Your setup: Pi 2B + university network

Specifics for exactly your hardware and situation — 2 Model B, HDMI screen available, laptop as your programming terminal, on a university network.

critical hardware fact

The Pi 2 Model B has no onboard Wi-Fi or Bluetooth — that only arrived with the 3B and Zero W onward. Your two options for getting it online are a wired Ethernet cable, or a USB Wi-Fi dongle plugged into one of its USB ports. This changes your whole setup plan, so it's worth deciding this first.

Ethernet vs USB Wi-Fi dongle — pick one

Ethernet (recommended for you)

Just works — no driver hunting, no authentication headaches, most reliable on a campus network. If you have an Ethernet port within cable reach (halls, library, lab), use this. Also the fastest and lowest-latency option for SSH work.

USB Wi-Fi dongle

More flexible location-wise, but adds a dependency: cheap dongles sometimes need extra drivers on Linux. Look specifically for one using the RTL8188 or RTL8192 chipset — both have solid built-in Raspberry Pi OS support with zero extra driver install.

Why university Wi-Fi is a different beast than home Wi-Fi

Home Wi-Fi is almost always WPA2-Personal — one shared password, and any device (including a headless Pi) connects instantly. University networks (often branded "eduroam") are usually WPA2/WPA3-Enterprise — instead of a shared password, every device authenticates individually with your university username and password through a protocol called 802.1X. This is why you can't just type the Wi-Fi password into Raspberry Pi Imager the way you would at home — the imager's simple Wi-Fi field is built for Personal networks, not Enterprise ones.

Connecting a headless Pi to eduroam-style networks

You configure it by hand, before first boot, by editing a file directly on the SD card's boot partition — or once booted, via the terminal. The core piece is a wpa_supplicant network block:

network={
    ssid="eduroam"
    key_mgmt=WPA-EAP
    eap=PEAP
    identity="your_uni_username@university.ac.uk"
    password="your_uni_password"
    phase2="auth=MSCHAPV2"
}

The exact eap/phase2 values depend on your specific university's configuration — check your IT department's eduroam setup page (search "[your university] eduroam Linux config") for the precise values, since some use PEAP+MSCHAPV2, others TTLS, and some require a downloaded root certificate.

the practical path for you

Since you already have an HDMI screen and keyboard: do first-time setup connected to the screen, not headless. Boot with monitor + keyboard attached, log in locally, and run the network setup (either plug Ethernet, or configure Wi-Fi through the desktop's network manager GUI or nmtui in the terminal — much easier than hand-editing config files blind). Once it's online and you've confirmed SSH works, disconnect the screen entirely and do all further work from your laptop over SSH. Screen only comes back out if something breaks.

A gotcha specific to campus networks: client isolation

Some university networks deliberately block devices on the same Wi-Fi from talking to each other directly (called AP/client isolation) — for security, to stop one student's malware reaching another's laptop. If your laptop can't SSH into the Pi even though both show as connected, this is the likely cause. Ethernet between them, or both on the same wired switch/socket, sidesteps this entirely — another reason Ethernet is the safer bet for your situation.

Your actual dev loop, put together

  1. 1
    Flash the SD card on your laptop
    Raspberry Pi OS Lite (Linux) via Raspberry Pi Imager, SSH enabled in the settings gear.
  2. 2
    First boot with HDMI screen + keyboard attached
    Get networking sorted here — Ethernet if possible, or Wi-Fi via nmtui/desktop GUI if using the dongle.
  3. 3
    Find its IP address
    ip addr show on the Pi itself, or check your router/campus network's connected-devices list.
  4. 4
    SSH in from your laptop
    ssh username@<ip-address> — from here on, the screen can come off and the Pi lives inside its case.
  5. 5
    Write and edit code on your laptop
    Either edit directly over SSH (nano/vim on the Pi), or edit locally and sync files over with scp/rsync, or use VS Code's "Remote - SSH" extension to edit Pi files with a full local-feeling editor.

The core — how a chip actually computes

Every board you've been looking at is really just this, scaled up: a switch, wired to millions of copies of itself, arranged so that patterns of on/off represent both data and instructions. Here's what's actually happening under that SoC when current flows.

to registers — sub-nanosecond, holds the current instruction to RAM — nanoseconds, holds active programs to storage — milliseconds, holds everything else to I/O — variable, the outside world

What that pulsing core is actually doing

Every single thing a CPU does breaks down into one repeating loop, running billions of times a second: fetch → decode → execute.

  1. 1
    Fetch
    The core reads the next instruction from memory — a plain number, stored at a specific address, that encodes what to do next.
  2. 2
    Decode
    Internal circuitry interprets that number: "this is an add", "this is a jump", "this is a memory read" — and figures out which parts of the chip need to switch on to carry it out.
  3. 3
    Execute
    The relevant circuitry actually does it — adds two numbers, moves data, changes what happens next — and the result lands somewhere (a register, memory, or an output pin), then the loop repeats.

The "glow" in the diagram above is a real physical thing, not just an animation choice: it's literally current moving through billions of transistors, each one flicking on and off in that fetch-decode-execute rhythm, at a rate set by the chip's clock — on a Pi 2B, 900 million times a second.

14

Computer engineering foundations

The layer beneath everything else in this poster — how "electricity" becomes "computation" in the first place.

Binary — why two states, not ten

A computer represents every number, letter, image, and instruction as sequences of two states: on and off (written as 1 and 0). Not because computers are bad at counting to ten — but because distinguishing two clearly separated voltage levels is far more reliable than distinguishing ten. Electrical noise, tiny voltage fluctuations, and manufacturing variance make a system with only two possible states vastly more robust than one trying to reliably tell apart ten different voltage levels billions of times a second.

The transistor — the one component everything is built from

A transistor is, at its simplest, a switch — but instead of a mechanical lever, it's switched by voltage. Apply voltage to one part (the gate), and it allows current to flow between two other parts (source and drain). No voltage, no flow. That's the entire primitive: a switch controlled by electricity instead of a finger.

The Pi 2B's SoC contains roughly 900 million transistors packed into a chip smaller than your fingernail, each one flicking on and off billions of times per second. Every capability of the board — every calculation, every pixel, every network packet — ultimately reduces to patterns of these switches flipping.

Logic gates — combining switches into decisions

Wire a handful of transistors together in specific patterns and you get logic gates — tiny circuits that implement basic decisions:

GateBehaviourReal-world analogy
ANDOutput on only if both inputs are onA door that needs two keys turned simultaneously
OROutput on if either input is onA light switched by two separate switches, either one works
NOTOutput is the opposite of the inputA "normally closed" relay — off input means on output
NANDAND followed by NOT — off only if both inputs are onThe universal building block — every other gate can be built from NAND gates alone

Chain enough of these together and you can build an adder (does arithmetic), then chain adders and memory elements together and you get an ALU (arithmetic logic unit — the part of the CPU that actually computes), and an ALU plus control circuitry plus memory access becomes a full CPU core.

Memory hierarchy — why there isn't just one kind of memory

You've already met three tiers of this on your own board. The pattern behind it: faster memory is dramatically more expensive per byte, so real systems use a pyramid — a small amount of very fast memory, backed by more of something slower, backed by a lot of something slower still.

TierSpeedSizePersists on power loss?
RegistersSub-nanosecondA few dozen valuesNo

Buses and the clock — how parts stay in sync

A bus is a shared set of wires that different components use to pass data — the address bus says "where", the data bus says "what". A clock is a signal that ticks at a fixed rate (the SoC's GHz rating), and every component agrees to only change state on a tick — this is what keeps millions of independent switching circuits synchronized instead of racing each other into garbage results. It's the same reason a rowing crew has a coxswain calling the stroke rate — individually-timed effort is chaos; a shared beat makes coordinated work possible.

Von Neumann architecture — the shape almost every computer follows

Named after mathematician John von Neumann, this is the architectural pattern your Pi (and your laptop, and almost everything with "computer" in the name) uses: a single memory space holds both the program's instructions and its data, and the CPU fetches from that same memory to figure out both what to do and what to do it to. The alternative (Harvard architecture — separate memory for instructions vs data) shows up mostly in small embedded microcontrollers where speed and simplicity matter more than flexibility.

Materials — why this, not that

Every material choice on the board and in a build like yours was an engineering trade-off, not a default. Here's the actual reasoning behind each one.

Silicon — the chip itself

Silicon is a semiconductor: pure, it barely conducts — but adding tiny, precise impurities (called doping, using elements like phosphorus or boron) creates regions that conduct in controllable, switchable ways. That controllability is the entire basis of the transistor. Silicon wins over alternatives (germanium was an early rival) because it's abundant (literally sand), and — crucially — it naturally forms a stable, insulating oxide layer (silicon dioxide, essentially glass) when exposed to oxygen, which is exactly what's needed to precisely insulate microscopic regions of a chip during manufacturing.

Copper — the traces

Copper isn't the best conductor (silver is slightly better), but it's roughly 100x cheaper and doesn't oxidize into a resistive layer nearly as much as most alternatives, while still conducting extremely well. For a board with a thousand traces, that combination of "excellent enough" conductivity and low cost at scale beats "slightly better" every time. Gold and silver get reserved for tiny, high-value spots — see below.

FR4 fiberglass — the board substrate

The rigid material under the copper (and under that green solder mask) is FR4: woven fiberglass cloth bonded with epoxy resin. It's chosen for a specific combination: electrically insulating (so copper traces don't short to each other through the board), mechanically rigid (doesn't flex and crack traces), flame-retardant (the "FR" — won't sustain a fire if something shorts and overheats), and cheap to mass-produce. Flexible PCBs exist (using polyimide instead) for applications needing bendable circuits, but they cost significantly more.

Gold plating — connector contacts

Look closely at USB port contacts, SD card edge connectors, or the "notches" some boards use for identification — a thin gold plating. Gold is a worse conductor than copper, and vastly more expensive — but it's used only in microscopically thin layers, and only where it matters: gold essentially doesn't corrode or oxidize. A connection you plug and unplug repeatedly needs a contact surface that stays clean and low-resistance for years; copper alone would gradually oxidize at that exposed contact point and start causing intermittent connection failures.

Solder — joining components to the board

Modern solder is almost universally lead-free now (tin-silver-copper alloys, common shorthand "SAC"), a shift driven by RoHS regulations banning lead in consumer electronics due to its toxicity. The trade-off: lead-free solder has a higher melting point and is slightly less forgiving to work with by hand than old tin-lead solder — which is why hobbyist soldering irons need to run hotter than they did a generation ago, and why lead-free joints can look slightly duller/grainier than the shiny tin-lead joints in old electronics.

Aluminum vs copper — heatsinks

Copper conducts heat better than aluminum, but aluminum is roughly a third of the weight and a fraction of the cost. For a small board like a Pi, the SoC doesn't generate enough heat to need copper's extra performance — so cheap, light aluminum heatsinks are the standard choice. High-end applications (GPUs, server CPUs) that generate far more heat often do pay copper's premium, sometimes as a copper baseplate under an aluminum fin stack — the best of both, where it's worth the cost.

ABS vs polycarbonate vs PLA — enclosures

ABS is the classic injection-molded case plastic: tough, impact-resistant, handles moderate heat well — most commercial Pi cases are ABS or a similar engineering plastic. Polycarbonate is stronger and more heat-tolerant still (and often clear), at higher cost — used where you want to see inside or need extra durability. PLA, the plastic most home 3D printers use by default, is biodegradable, easy and cheap to print, but softens at a relatively low temperature (~60°C) — fine for a decorative or lightly-used case, risky for anything that sits near a heatsink or in direct sun. If you 3D print your cyberdeck's enclosure, this is exactly why people upgrade to PETG (tougher, more heat-resistant than PLA, still easy to print) for anything electronics-adjacent.

Ceramic and plastic IC packages

The material encasing a chip (its "package") is chosen for how it manages heat and protects the silicon inside. Cheap consumer chips use plastic packages (cost-optimized, adequate for low heat). Higher-performance or extreme-environment chips use ceramic packages, which handle heat and mechanical stress better but cost significantly more to produce — another case of "good enough and cheap" winning for mass consumer hardware, with the premium material reserved for where it earns its cost.

Solder mask green — the one non-functional choice

Worth restating since it surprises people: the green color itself carries no electrical or functional benefit. It's a historical/economic default — green solder mask pigment was cheap and well-understood early in PCB manufacturing, and the industry standardized around it. Every other color (red, blue, black, purple) performs identically; it's purely aesthetic once you're past "does the mask insulate and resist solder properly," which they all do.

the pattern underneath all of these

Almost every material choice on this board follows the same logic: use the cheapest material that clears the bar for a given job, and reserve expensive, high-performance materials only for the specific spots where their extra property (corrosion resistance, thermal conductivity, heat tolerance) is actually load-bearing. Recognizing that pattern is most of what "engineering intuition" for material selection actually is.

Numbers & data — what a bit actually is

Section 14 covered binary as electrical states. This is the next layer up: how those on/off states become numbers, letters, colors, and sound.

Binary, decimal, hex — same number, three costumes

A single binary digit (bit) is 1 or 0. Group 8 of them and you get a byte — 256 possible patterns (0 to 255). Hexadecimal (base 16, digits 0-9 then A-F) exists purely for human convenience: one hex digit represents exactly 4 bits, so a byte is always exactly 2 hex digits. That's why you see hex everywhere in computing — memory addresses, color codes, MAC addresses — it's a compact, exact shorthand for binary that decimal can't offer (a byte in decimal is an awkward 0-255, not a clean digit count).

BinaryDecimalHex

That teal color you've been looking at all through this poster, #5DCAA5 — that's literally three bytes in hex: 5D red, CA green, A5 blue, each 0-255, mixed together as light.

Text — ASCII and Unicode

A letter is just a number with an agreed-upon meaning. ASCII, the original standard, maps numbers 0-127 to English letters, digits, and punctuation — 'A' is 65, 'a' is 97. This is why so much old computing was English-centric: 127 values simply couldn't fit every alphabet on Earth. Unicode replaced this with a scheme that can represent over a million characters — every language, plus emoji — with UTF-8 (the encoding you'll see everywhere now) cleverly staying backward-compatible with ASCII for the first 128 values while extending outward for everything else.

Images — pixels as numbers in a grid

A digital image is a grid of pixels, and each pixel is just 3 (or 4, with transparency) byte-sized numbers — red, green, blue intensity. A 1920×1080 image is, underneath, a flat list of roughly 6 million bytes. "Compression" (JPEG, PNG) is just cleverer, smaller ways of describing that same grid — patterns and redundancy encoded more efficiently than storing every pixel's raw value.

Sound — sampling a wave into numbers

A microphone produces a continuous electrical wave. To store it digitally, the system measures (samples) that wave's height thousands of times per second — CD-quality audio takes 44,100 samples every second, each stored as a number. Play those numbers back through a speaker fast enough, in order, and your ear reconstructs something close enough to the original wave to sound continuous. This is exactly the same principle as PWM from Section 09, just running in the opposite direction — sampling analog into digital instead of approximating analog from digital.

the single idea underneath this whole section

There is no separate "type" of data at the hardware level. A photo, a song, this webpage, and a running program are all, physically, just long sequences of bits sitting in memory or storage. What makes one a "photo" and another "code" is entirely about which program is interpreting those bits, and what rules it applies to them.

Operating systems — Linux under the hood

You're running Raspberry Pi OS, which is Linux — here's what that word is actually doing for you every time you type a command.

Kernel vs userspace

The kernel is the innermost layer of the OS — the only piece of software allowed to talk to hardware directly (memory, GPIO, disks, network cards). Everything else you run — your Python script, the terminal, ls, a web browser — is userspace, and has to go through the kernel (via defined "system calls") to do anything that touches real hardware. This separation exists so one crashing or misbehaving program can't take the entire machine down or reach into memory it shouldn't — the kernel enforces the boundary.

Processes and threads

A process is a running program with its own private memory space — when you run python3 main.py, that's one process. A thread is a unit of execution inside a process — a process can have multiple threads running concurrently, sharing that process's memory. The kernel's scheduler is what makes a single-core Pi 2B feel like it's running many programs "at once": it rapidly switches the CPU core between processes, giving each a tiny time slice, fast enough that the switching itself is imperceptible.

The filesystem and permissions

Linux organizes everything — including devices — as files, from a single root /. /home/sabrine is your files, /etc holds system configuration, /dev even represents hardware devices as files you can (carefully) read/write directly. Every file has an owner and permission bits (read/write/execute, for owner/group/everyone else) — this is what sudo is bypassing when you use it: normal commands run as your limited user account; sudo temporarily elevates to the root superuser, who can touch anything. Using it casually is how systems get broken — treat it as "I am now allowed to make an unrecoverable mistake," not as a formality.

ls -l main.py
# -rwxr-xr-x  1 sabrine sabrine  842 Jul 20 14:02 main.py
# permissions  owner   group     size    date       name
# r = read, w = write, x = execute — shown for owner / group / others, in that order

Why Raspberry Pi OS specifically is Debian-based Linux

"Linux" strictly refers only to the kernel — the rest (command-line tools, package manager, default software) comes from a "distribution" built around that kernel. Raspberry Pi OS is built on Debian, one of the oldest and most stable distributions, which is why its package manager is apt (apt install, apt update) — the same tooling you'd meet on Ubuntu, since Ubuntu is itself Debian-based. Learning apt here transfers directly to a huge share of the Linux world.

systemd, revisited properly

You already used this in Section 10 to keep your script running. What it actually is: the first userspace process the kernel starts (process ID 1), responsible for starting every other service in the right order, restarting things that crash, and managing the boot sequence's final stages. It's controversial in the Linux world (some distributions deliberately avoid it in favor of simpler alternatives) precisely because it does so much — but on Raspberry Pi OS and most modern distributions, it's the standard.

Networking — how SSH actually works

You've been using SSH this whole poster. Here's what's actually happening underneath ssh username@ip-address.

IP addresses and ports

An IP address (like 192.168.1.42) identifies a specific device on a network — like a street address. A port is a number (0-65535) identifying which specific service on that device you want to reach — like an apartment number at that address. SSH conventionally lives on port 22; a web server on port 80/443. One device, one IP, but potentially dozens of different services simultaneously reachable at different ports.

DNS and why .local sometimes fails

DNS translates human-readable names (google.com) into IP addresses. raspberrypi.local uses a smaller-scale cousin of this called mDNS (multicast DNS) — devices announce their own name directly on the local network instead of asking a central server. This is exactly the mechanism that can get blocked by university network isolation settings (Section 12) — another reason a raw IP address is a more reliable fallback than the .local hostname on a campus network.

TCP — why data arrives intact

Data crossing a network gets broken into packets, which can arrive out of order, duplicated, or not at all. TCP (Transmission Control Protocol, which SSH runs on) handles this: it numbers every packet, has the receiver acknowledge what arrived, and automatically re-sends anything missing — so the application on top never has to think about packet loss at all, it just sees a clean, ordered stream of bytes. UDP is the leaner alternative that skips all that guarantee-checking — used where speed matters more than perfection (video calls, some game traffic), since re-requesting a lost data isn't worth the delay it'd cause.

What SSH actually does when you connect

  1. 1
    Key exchange
    Your laptop and the Pi negotiate a shared secret using public-key cryptography, without ever sending that secret itself over the network — an eavesdropper watching the traffic can't reconstruct it even having seen the whole exchange.
  2. 2
    Encryption established
    From this point, everything sent between laptop and Pi is encrypted with that shared secret — your password, your commands, your file transfers are all unreadable to anything intercepting the traffic.
  3. 3
    Authentication
    You prove who you are — either a password, or (better) an SSH key pair, where your laptop holds a private key and the Pi holds the matching public key, and you never type a password over the network at all.
  4. 4
    A shell, remotely
    Once authenticated, the Pi starts a normal shell process for you, but its input/output are piped over that encrypted connection instead of a physical keyboard/screen — which is why it feels exactly like sitting at the Pi directly.

Setting up SSH keys (worth doing properly)

# On your laptop, generate a key pair once:
ssh-keygen -t ed25519 -C "sabrine-laptop"

# Copy the public key to the Pi:
ssh-copy-id sabrine@<pi-ip-address>

# From now on: ssh sabrine@<pi-ip-address>  logs in with zero password typing

Beyond convenience, this is meaningfully more secure than password auth — there's no password to guess or leak, and you can later disable password login entirely on the Pi so only devices holding the matching private key can ever connect.

Languages — what "running code" means

You've written Python for GPIO and C for pigpio in this poster already. Here's what actually happens between you typing code and the LED turning on.

Compiled vs interpreted

A CPU only ever executes one thing: machine code — raw binary instructions specific to its exact architecture (the Pi's is ARM). Source code you write has to become that somehow, and there are two broad strategies:

Compiled (C, C++, Rust)

A compiler translates your entire source file into machine code once, ahead of time, producing a standalone executable. Running it later means the CPU executes that machine code directly — fast, but you must recompile after every change, and the result only runs on the CPU architecture it was compiled for.

Interpreted (Python, in practice)

Python source is actually first compiled to an intermediate form called bytecode (you'll see this cached as .pyc files), then the Python interpreter reads and executes that bytecode step by step at runtime. This adds overhead compared to running raw machine code directly — which is exactly why the timing-critical pigpio work in Section 08 reaches for a compiled C library instead of pure Python.

Why your Python GPIO calls end up touching real pins

When you call GPIO.output(17, GPIO.HIGH), that Python call doesn't touch hardware directly — it calls into a C extension (RPi.GPIO is actually written in C underneath its Python interface) which makes a system call into the kernel, which has the actual driver code allowed to write to that physical pin's memory-mapped register. Every layer here — your Python, the C extension, the kernel, the hardware register — is a real, separate boundary; "programming the GPIO" is really "asking the kernel, through several layers of translation, to flip one bit in a hardware register."

The build-and-run cycle for C, made concrete

gcc file.c -lpigpio -lrt -pthread -o out
# ↑ this line: source code → machine code, done once
./out
# ↑ this line: CPU executes that machine code directly, no translation layer

Compare that to Python, where python3 main.py both translates and runs in the same step, every single time you launch it — one more reason Python feels more immediate to iterate on, and one more reason it's slower per-instruction than compiled code.

Windows, macOS, Linux — the whole landscape

Every "what OS should I use" conversation, and what the guys around you actually mean when they throw these names around.

What an operating system's job actually is, stated plainly

Strip away the branding and every OS does the same core job (Section 17 covered the mechanics): manage hardware, run programs, keep them from stepping on each other, give you a way to interact with it. Windows, macOS, and Linux differ in who controls the code, who it's designed for, and what philosophy shaped its defaults — not in whether they can technically do the job.

Windows

Made by Microsoft, closed-source, dominant on consumer/business desktops and gaming. Its history goes back to DOS; it prioritizes broad hardware/software compatibility and a consistent GUI experience. PowerShell and cmd (its terminals) exist here — see Section 22.

macOS

Made by Apple, closed-source, runs only on Apple's own hardware. Under the hood it's actually Unix-based (built on a foundation called Darwin, related to BSD) — which is exactly why macOS's terminal feels much closer to Linux's than to Windows's: ls, grep, ssh all work natively.

Linux

Not a company's product — an open-source kernel (Section 17) that anyone can use, modify, and redistribute, free of charge. This is why it "keeps getting thrown around": it isn't one OS, it's the shared foundation under an entire ecosystem of different ones.

So what is Linux, exactly, and why does everyone say it differently

Linus Torvalds wrote the Linux kernel in 1991 and released it under a license (GPL) that lets anyone use and modify it freely, as long as they share their changes back. That single decision is why Linux is everywhere you don't see it: it runs most of the world's servers and cloud infrastructure, every Android phone (Android's kernel is Linux), most supercomputers, and your Raspberry Pi. Different groups then wrapped that same kernel with different tools, defaults, and philosophies, calling the result a distribution ("distro") — which is why "Linux" sounds like it means five different things depending on who's talking.

CPU architectures — x86, ARM, and 32 vs 64-bit

Why your laptop and your Pi are running fundamentally different "dialects" of machine code, and what that number after "-bit" means.

Instruction sets — the actual dialect difference

Section 19 established that a CPU only executes machine code, and that machine code is architecture-specific. "Architecture" here means the instruction set — the exact vocabulary of operations a chip understands. Compiled code for one architecture is gibberish to a chip built for another; this is the real, physical reason an app compiled for your laptop can't just run on your Pi unmodified.

x86 (and x86-64)

Intel and AMD's architecture — what's inside almost every Windows/Linux laptop and desktop. A CISC design (Complex Instruction Set Computing): individual instructions can do fairly complex things in one step, at the cost of more complicated, power-hungrier chip internals to decode them.

ARM

What your Pi runs (and every phone, and Apple Silicon Macs since 2020). A RISC design (Reduced Instruction Set Computing): individual instructions are deliberately simple, so the chip decoding them can be smaller, cheaper, and dramatically more power-efficient — which is exactly why ARM dominates anything battery-powered or thermally constrained, from phones to your Pi.

RISC-V

A newer, fully open-source instruction set (unlike ARM and x86, which are licensed/proprietary) gaining serious traction in research and increasingly commercial chips — worth knowing the name, since it's the one genuinely new entrant challenging the ARM/x86 duopoly.

Why Apple switching to ARM was a big deal

Apple's move from Intel (x86) to their own ARM-based chips (2020) was industry-shaking specifically because it proved RISC-style efficiency could match or beat x86 performance even outside phones — undermining the old assumption that "real" laptop/desktop power required CISC.

32-bit vs 64-bit — what the number actually measures

It refers to the width of a CPU's registers and memory addresses — how large a single number it can natively handle, and how much memory it can directly address. A 32-bit system can address at most 2^32 bytes (4GB) of memory — which is exactly why old 32-bit Windows famously couldn't "see" more than 4GB of RAM even when more was installed. 64-bit systems (essentially everything now, including current Raspberry Pi OS) address 2^64 bytes — a number so large it's not a practical ceiling for the foreseeable future.

tying it back to your board

Your Pi 2B's SoC is a 32-bit ARM chip (Cortex-A7) — later Pi models moved to 64-bit ARM cores. This is a real, practical constraint worth knowing: it caps addressable RAM around 4GB (moot on a 2B which only has ~1GB anyway) and means some newer 64-bit-only OS images won't run on your specific board — always check a distro explicitly supports "Pi 2" or "armhf/32-bit" before flashing it.

The engineer's canon — what's actually non-negotiable

Everything above builds fluency in how machines work. This is the separate list of practical skills that mark someone as operating at a working engineer's level, regardless of specialty.

Version control — git

The single most universal tool in all of software/hardware engineering — a system for tracking every change ever made to a project's files, who made it, and why, with the ability to branch off, experiment, and merge back safely. Not knowing git is one of the fastest ways to be underestimated in a technical room; knowing it well is one of the fastest ways not to be.

git init                          # start tracking a project
git add .                         # stage changes
git commit -m "message"           # save a snapshot with a description
git remote add origin <url>       # link to a remote copy (e.g. GitHub)
git push                          # send your commits there
git clone <url>                   # copy someone else's project down

Data structures & Big-O — how to reason about "fast" and "slow"

A data structure is just an organized way of storing data (an array, a list, a tree, a hash table) — and the right choice changes how expensive an operation is, sometimes by orders of magnitude. Big-O notation is the standard shorthand for describing how an operation's cost grows as data grows: O(1) constant (instant, regardless of size), O(n) linear (cost scales directly with size), O(n²) quadratic (gets painful fast), O(log n) logarithmic (barely grows at all, the good kind). This is the actual language used when engineers discuss whether an approach "scales."

Virtualization and containers

A virtual machine emulates an entire computer — full OS, virtual hardware — running inside another machine, isolated but heavy. A container (the concept behind Docker) is lighter: it shares the host's kernel but isolates everything else (filesystem, processes, network) — giving most of the isolation benefit of a VM at a fraction of the resource cost. This distinction is why "just Dockerize it" became a universal phrase — it's the practical, everyday tool for "make this run identically anywhere," which matters enormously once you're distributing software rather than running it on one machine you personally control.

Databases, the essential split

Relational (SQL) databases (PostgreSQL, MySQL) store data in strict tables with defined relationships between them — powerful for data with clear structure and where correctness/consistency matters most. NoSQL databases (MongoDB, Redis) trade some of that structure for flexibility and speed at scale — good for data that doesn't fit neatly into tables, or where raw throughput matters more than rigid consistency. Neither is universally "better" — it's a real trade-off engineers make per-project.

Security fundamentals

Encryption scrambles data so only someone with the right key can read it (what SSH does, Section 18). Hashing is a one-way street — it turns data into a fixed-length fingerprint that can't be reversed back to the original, used for things like storing passwords: a system should never store your actual password, only its hash, so even a full data breach doesn't hand over usable passwords. HTTPS is HTTP (the web's core protocol) wrapped in the same kind of encryption SSH uses — that padlock icon in your browser is that encryption handshake having succeeded.

the real point of this section

Nobody holds all of this fluently on command — what actually separates confident engineers from everyone else in a room is knowing the shape of each idea well enough to ask a sharp question or follow a technical conversation, and knowing precisely which of these to go deep on for your own work. For you specifically, right now: git is worth genuine fluency immediately (you'll use it on every project from here on), and the rest are worth exactly what this section gave you — the map, so nothing said around you sounds like a foreign language.

ARM assembly — the layer beneath your Python

Section 19 said compiled code becomes machine code. Assembly is machine code's human-readable face — the lowest level anyone actually writes by hand, one step above the raw 1s and 0s themselves.

What assembly actually is

Machine code is just numbers — an ARM instruction might literally be the bit pattern 1110 0001 1010 0000.... Nobody writes that directly. Assembly language gives each instruction a short mnemonic name instead (ADD, MOV, LDR), and an assembler does the pure, direct, 1-to-1 translation from that mnemonic back into the exact bit pattern — no cleverness, no optimization, unlike a compiler translating from C or Python. This is the closest a human ever gets to speaking the CPU's actual language.

Registers — the CPU's only "hands"

Section 14 introduced registers as the fastest, tiniest tier of memory. On ARM, there are 16 general-purpose registers, named r0 through r15, each holding one 32-bit value (on your 2B's 32-bit chip). A few have reserved jobs: r13 is the stack pointer, r14 holds the return address after a function call, r15 is the program counter — literally the address of the instruction currently executing. Every single operation in assembly is some variation of "move a value into a register," "do arithmetic between registers," or "read/write a register's value to memory."

A real program: adding two numbers

.global _start

_start:
    mov r0, #5          @ put the number 5 into register r0
    mov r1, #7          @ put the number 7 into register r1
    add r2, r0, r1      @ r2 = r0 + r1  →  r2 now holds 12

    @ exit the program cleanly, using r2 as the exit code
    mov r7, #1          @ syscall number 1 = exit
    swi 0                @ software interrupt — hand control to the kernel
# assemble, link, and run it on your Pi:
as -o add.o add.s
ld -o add add.o
./add
echo $?             # prints 12 — the exit code we set in r2

Notice the last two instructions: swi 0 is a software interrupt — the literal, physical mechanism a userspace program uses to hand control to the kernel (Section 17's kernel/userspace boundary, made concrete). This is what a "system call" actually *is* at the hardware level — not a metaphor, an actual instruction that exists for exactly this purpose.

Why this matters even though you'll basically never write it by hand

You will almost certainly write Python and C for the rest of your work, not assembly — that's normal and correct, compilers do this translation far better and faster than a person should by hand for real projects. What this section actually buys you: when someone mentions "the stack overflowed," "a branch misprediction," or "this is a syscall," you now know those aren't jargon, they're literal, physical events at exactly this layer — r13 running out of room, a B instruction guessing wrong about which way an if would go, or an swi instruction firing. That's the difference between memorizing terms and actually understanding the machine underneath them.
